### 명령형? 선언형?

명령형 프로그래밍과 선언형 프로그래밍은 언어에서 정해지기도 하지만 평소 프로그래머가 어떻게 프로그래밍을 하는 그 방식에 따라서 나뉜다고 생각합니다.

> __명령형 프로그래밍은 과정(어떻게?)를 중시한다.__

> __선언형 프로그래밍은 결과(무엇을?)을 중시한다.__

그 예를 살펴보겠습니다.

#### 1. Red Lobster

나는 회사에서 열심히 일하고 퇴근한 후 Red Lobster에서 외식을 하기로 했다. 식당에 도착한 후 안내 데스크에 가서 이야기를 한다.

> __명령형 : 12번 테이블 자리가 비어 있습니다. 나와 우리 가족은 저곳에 걸어라 앉을 것입니다.__

> __선언형 : 4명 앉을 자리 부탁해요.__

위에서 말했듯이 명령형 방식은 자리에 어떻게 도착해서 앉을 것인지에 대한 과정을 중시하고 선언형은 4명이 앉을 자리를 말하는데에서 결과를 중시하는 것을 알 수 있습니다.


#### 2. 집 주소 물어보기

다른 예시로 누군가의 집 주소를 물어본다고 가정합시다.

"당신의 집 주소는 어디인가요?"

> 명령형: 여기서 직진 후 우회전 한 다음 500m 앞 블록에서 좌회전 후 빵집 앞에서 좌회전 하시면 나오는 22번로 109동이 저의 집입니다.

> 선언형: 제 주소는 xx시 xx동 xx 22번로 109동입니다.

다음 예시에서도 명령형은 그 집에 도착하는 과정을, 선언형은 그 집의 도착할 주소를 중시하는데서 과정과 결과를 각각 중시한다고 생각할 수 있습니다.

### 명령형 언어? 선언형 언어?

선언형 언어의 대표적인 예시로는 CSS, HTML 등이 있습니다.

HTML은 프로그래밍 언어가 아니라는 이야기는 모두 한번씩은 들어봤을 것입니다. 
이 부분은 생각하는 사람의 관점에 따라서 달라지긴 하겠지만 
C나 Python과 직접 프로그래밍을 하는 명령형 언어와는 다르다는 것을 느낄 수 있습니다.
이 것이 바로 명령형 언어와 선언형 언어의 차이점에서 나온 이야기입니다.

HTML은 브라우저가 어떻게 block들을 배치하고 나타낼지에 대한 그 결과만 적어둘 뿐
이것을 각 브라우저들이 어떻게 파싱할지에 대한 정보는 담고 있지 않습니다.

CSS 또한 디자인이 어떻게 나타날지에 대한 결과에 대한 정보만 담고 있을 뿐
그 중간 과정을 처리하는데는 각 브라우저마다 다를 정도로 과정은 담고 있지 않습니다.

---

### 명령형 프로그래밍? VS 선언형 프로그래밍?

앞서 계속 말했던 것으로 명령형은 과정을, 선언형은 그 결과를 중시하는 것을 의미합니다.
그렇다면 명령형 프로그래밍과 선언형 프로그래밍 중 무엇이 더 좋은 걸까요?

그에 대한 저의 생각은 "상황에 따라서 다르다." 입니다.

예시를 들어 보겠습니다.

```
// 배열을 파라미터로 받고 각 요소들의 값에 2를 곱하는 함수
// 함수 1
function double (arr) {
  let results = []
  for (let i = 0; i < arr.length; i++){
    results.push(arr[i] * 2)
  }
  return results
}
```

```
// 함수 2
function double (arr) {
  return arr.map((item) => item * 2)
}
```

함수 1과 함수 2는 JS를 다뤄본 사람에게는 좀 더 익숙하겠지만 다른 언어를 사용하셨던 분들도 얼추 어떤 함수인지는 알 수 있을 것입니다.(혹시 몰라서 이야기하면 둘은 같은 함수입니다.
)

1번의 경우 어떻게 각 요소들의 값이 변할 것인지에 대한 과정을 일일이 프로그래밍한 명령형 프로그래밍입니다.
2번의 경우 JS가 기본적으로 제공하는 map함수를 이용하여 만든 결과 값을 얻기 위한 선언형 프로그래밍입니다.

선언형 프로그래밍을 이용하면 그 시간을 단축시킬 수 있을 것입니다.
하지만 선언형 방식으로만 프로그래밍을 하게 되면 사용하고 있는 프로젝트의 구조를
이해하지 못해 추후 문제가 생길 수도 있습니다.

대표적인 예시로 리액트는 JSX 문법을 이용해서 HTML에 어떻게 블록을 나타낼지에 대해서
선언하는 프로그래밍 방법입니다. 리액트를 이용할 때 어떻게 HTML에 반영될지에 대한
과정은 저희가 따로 손보지 않고 리액트 라이브러리가 처리해줍니다. 이 부분이 편리하긴 하지만
최종적인 리액트를 이용한 최적화 방법에는 리액트에 대한 구조적인 이해가 필요할 것이고 이 부분에서 요구되는 것이
명령형 프로그래밍 방식일 것입니다.

이 부분에 대해서는 추후 리액트에 대한 글을 쓰면서 다시 정리하겠습니다.



---

## 결론

선언형 프로그래밍은 더 빠른 프로그래밍을 가능하게 하지만 명령형 프로그래밍의
기반이 없으면 프로젝트의 구조에 대한 지식에서 구멍이 생길 수도 있으니
사용하고 있는 라이브러리, 프레임워크들이 어떻게 작동하는지에 대한
지식을 알고 있어야합니다.
