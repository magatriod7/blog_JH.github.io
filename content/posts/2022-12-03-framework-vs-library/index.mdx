모두 프로그래밍을 하다보면 프레임워크와 라이브러리에 대해서 한번쯤은 들어보셨을겁니다.

[위키백과](https://ko.wikipedia.org/wiki/%EC%86%8C%ED%94%84%ED%8A%B8%EC%9B%A8%EC%96%B4_%ED%94%84%EB%A0%88%EC%9E%84%EC%9B%8C%ED%81%AC)에 따르면 프레임워크는 컴퓨터 프로그래밍에서 소프트웨어 프레임워크(software framework)는 복잡한 문제를 해결하거나 서술하는 데 사용되는 기본 개념 구조이다. 간단히 뼈대, 골조(骨組), 프레임워크(framework)라고도 한다. 라고 정의합니다.

여기서 눈여겨봐야할 것은 골자와 뼈대입니다. 어플리케이션 개발을 할 때 프레임워크는 구동, 메모리, 이벤트 루프 등을 관리하게 됩니다. 사용자는 프레임워크가 정해준 방식대로 클래스와 메서드를 구현합니다.

대표적인 예시로는 Spring(Java), Django(Python), android, Cocoa Touch, Angular(JS), Vue.js(Js)등이 대표적으로 있겠네요.

[위키백과](https://cocoon1787.tistory.com/745) 라이브러리의 정의는 라이브러리(영어: library)는 주로 소프트웨어를 개발할 때 컴퓨터 프로그램이 사용하는 비휘발성 자원의 모임이다. 여기에는 구성 데이터, 문서, 도움말 자료, 메시지 틀, 미리 작성된 코드, 서브루틴(함수), 클래스, 값, 자료형 사양을 포함할 수 있다. OS/360 및 이후 세대에서는 파티션 데이터 세트로 부른다. 라고 합니다.

쉽게 말해서 특정 기능을 사용하기 위해서 따로 모아둔 소스코드들의 모임 정도로 생각할 수 있겠네요.

대표적인 예시로는 tensorflow(Python), STL(C++), jQuery(JS) 등이 있습니다.

다른 예시를 들어보면 라이브러리는 집을 짓는것과 같습니다. 집을 저희가 원하는 스타일대로 만들 수 있으면 배치할 수 있습니다.

반면 프레임워크는 새 집을 사는 것과 같습니다. 집의 배치를 바꿀 수 없겠죠.

### 정의

라이브러리는 개발자에게 미리 정의된 기능과 클래스를 제공하여 작업을 쉽게 만들어준다.

프레임워크는 개발자가 특정 플랫폼용 애플리케이션을 구축하는 골자, 토대가 된다.

### 제어의 역전

라이브러리를 사용하여 애플리케이션의 흐름을 제어하고 라이브러리를 호출 할 수 있습니다.

프레임워크를 사용하면 프레임 워크가 흐름을 제어하고 코드를 호출합니다.

### 구성

라이브러리는 일반적으로 모듈, 객체, 함수, 클래스 등으로 이루어진 코드 모음입니다.

프레임워크는 많은 API, 컴파일러, 도구, 라이브러리 등으로 구성됩니다.

### 코드 수정

라이브러리 코드는 특정 상황을 해결하기 위해 만들어졌기 때문에 필요에 따라 라이브러리 코드를 수정할 필요가 있습니다.

프레임워크는 사용자가 코드를 수정할 수 없습니다.

### 범위

라이브러리는 어디서나 호출이 가능합니다.

프레임워크는 정해진 상황에서만 호출이 가능합니다.

### 성능

라이브러리는 비교적 가볍습니다.

프레임워크는 비교적 무겁습니다.

### 확장성

기본적으로 라이브러리는 확장성을 위해 설계되지 않습니다.

프레임워크는 일반적으로 확장성을 고려합니다.

### 교체가능

기존의 라이브러리가 마음에 들지 않으면 다른 라이브러리를 선택하는 것이 비교적 자유롭습니다.

프레임워크는 교체하기 어렵습니다.

### 장점

라이브러리: 재사용성, 제어, 속도

프레임워크: 지원, 빠른 프로그래밍

## 리액트는 라이브러리인가? 프레임워크인가?

그렇다면 리액트는 프레임워크일까요 아니면 라이브러리일까요? 결론만 말하자면 둘의 성격을 모두 가진다가 맞겠네요.

리액트 공식 홈페이지에서는 라이브러리라고 했지만, 이 부분은 많은 사람들이 토론을 하는 주제입니다. 정해진 답은 없다고 보는 것이 맞습니다.

이제 한번 따져볼까요?

### 리액트의 제어의 역전

프레임워크와 라이브러리의 대표적인 차이점 중 하나는 `제어 흐름`의 권한이 어디에 있는가입니다.

저희가 흔히 사용하는 라이브러리의 경우 저희가 원하는 곳에서 해당 라이브러리를 호출하여 코드를 사용할 수 있습니다.
반면 프레임워크를 이용하기 위해서는 프레임워크가 짜 놓은 틀 안에서 코드를 작성해야합니다.

즉, 프레임워크가 저희의 애플리케이션 흐름 제어 권한을 가져갑니다. 이를 `제어의 역전`이라고 합니다.

리액트는 기본적으로 제어의 전체적인 흐름이 어느정도 정해져 있다고 생각합니다. hook의 경우, useState로 이벤트를 관리, state로 상태를관리, render을 이용해 렌더링할 요소들을 정합니다.
제어의 흐름을 리액트가 정하니까 리액트는 프레임워크의 성격을 띄고 있군요?라고하면 맞는 말이지만 리액트는 프레임워크군요? 라고 하면 애매해집니다.

프레임워크와 같이 어느정도 형식을 띄고 있는 것은 맞지만 결국 이벤트의 실행과 제어는 사용자가 특정 부분에서 일어나도록 정할 수 있어 사용자가 제어의 흐름을 가지고 있기 때문입니다.

이를보고 라이브러리의 성격을 가진 프레임워크인게 아닐까요? 라고 하면 그것도 맞는 견해일 것입니다. 앞서 말했듯이 이 부분은 아직도 많은 토론을 하는 분야니까요.

### 리액트의 구성

개인적으로 이 부분은 라이브러리가 맞다고 생각합니다. 리액트를 실행하기 위해서는 기본적으로 JSX 컴파일러가 필요합니다. 대표적으로 babel이 있겠네요.

리액트 내부에서 추가적인 기능을 사용할 경우 대표적인 예시로 react-player와 같은 다른 사람들이 만든 라이브러리를 호출하는 경우가 많이 있을 것입니다.

앞서 말한대로 프레임워크는 컴파일러, 라이브러리, API등을 프레임워크가 자체적으로 제공하는 것에 반해서 리액트는 리액트를 작동시키는 것만 할 뿐 컴파일러와 같은 기능은 제공하지 않는 부분에서 라이브러리의 성격을 가지고 있습니다.

### 코드 수정

이 부분은 프레임워크에 가깝다고 생각합니다. 기존 정해져 있는 리액트의 틀을 수정할 생각을 할 사람은 별로 없을 것이라고 생각합니다.

### 주관적인 결론

앞서 말한 부분 외에도 따질 수 있는 부분은 많겠지만 프레임워크의 의의가 개발의 골자와 뼈대를 만드는 것이고 리액트는 특정 기능을 위한 소스코드의 모임이라는 점을 생각할 때,
리액트는 프레임워크라고 하기에는 컴파일러나 내부 제공 API 등이 적다고 생각합니다.

그 점에서 저는 프레임워크의 성격을 가지고 있는 라이브러리라고 생각합니다.
